spring-boot启动解读

Jul 16, 2017Spring Boot Hits留言

1、背景

spring-boot可以采用java -jar embedded-container-1.0-SNAPSHOT.jar类似这样的方式直接启动应用，我们在这个过程中有几个问题我觉得是可以探究的：

  • 1、spring-boot是如何打包的，只是把我们写的main函数写进了MANIFEST.MF中的main-class吗？
  • 2、它是如何知道我们依赖哪些starter，并且需要注入哪些bean（starter）？
  • 3、它是怎么处理内置容器和应用之间的关系的？
  • 4、我们能用这种机制做哪些扩展？

做这样的解读最终的目的还是我想说的第四点。

2、spring-boot如何打包

先看package，由于我们在pom中依赖了springboot的plugin，所以在打出来的内容中会和普通打包有些差异：

可以看到会有两个同名的jar文件，其中一个带了original后缀。我们主要来看不带original的jar包，这个包就是spring-boot-maven-plugin打出来的jar，我们把它解压开来：

我们可以看到打包的内容里面有个目录叫BOOT-INF，这里面装的是我们自己写的类，而外部真正的执行程序却是org.springframework.boot包，在看看MANIFEST.MF文件，果然main-class指向的是spring-boot包里的JarLauncher:

从这个文件中不难看出，入口类是JarLauncher，但是真正程序的执行却是com.mogujie.embedded.container.Main类，这也给了我们一个很好的思路，如果我们要对启动做一些定制，可以采用这样的方式去接入，直接在打包的过程就把它做到对代码无侵入。我们先不看JarLauncher怎么执行，我们先看spring-boot是如何打出这个包的。
其实全仰仗maven的plugin机制：http://docs.spring.io/spring-boot/docs/current/maven-plugin/usage.html

3、如何启动

我们再来顺着JarLauncher看看spring-boot如何启动应用。
这里比较关键的是spring-boot的launch会自己创建一个classloader（LaunchedURLClassLoader extends URLClassLoader ）。这里也有一个archive的概念，当前创建的classpath中包含的就是BOOT-INF下的classes和lib下的所有类。spring-boot还支持reload和restart技术，加快加载速度，具体可以看：http://docs.spring.io/spring-boot/docs/current-SNAPSHOT/reference/htmlsingle/#using-boot-devtools-restart

真正的启动很简单，拿到我们自己程序的main方法进行invoke即可，当然classloader是采用当前线程已经创建的classloader：

由于我们自己的类都是在BOOT-INF下面，不在classpath下，所以很容易理解为什么要创建一个classloader。

4、如何加载starter

再接下来的问题就是，程序如何知道我们用了哪几个starter。为什么只要在pom中依赖，不需要在spring中配置就可以加载到对应的bean？
这里有个标签（annotation）很重要：
@EnableAutoConfiguration

具体的实现在org.springframework.boot.autoconfigure.EnableAutoConfigurationImportSelector类，我们可以具体看看，这个类做了什么：

这里就会加载到所有可以被auto config的类，通过这些configuration来加载对应的bean。程序会去加载META-INF下的spring.factories文件，如果扫到了
org.springframework.boot.autoconfigure.EnableAutoConfiguration
就会加载对应的bean进行处理，我们可以看下mogu-boot对第三方依赖的auto config:

原生spring-boot的配置是放在了spring-boot-autoconfig包下进行管理的：

所以我们要考虑的是采用这个机制和spring原生的去import一个xml文件，或者是maven依赖之后，再去依赖一个配置是否有本质上的区别？我觉得还是存在一些区别的：

  • 这种方式统一管理了接入方的规范要求，让第三方的接入可以按照一定的标准来执行，这样的方式是非常有利于共享共建的。
  • 配置管理和依赖管理的分离，这样的好处是，我只需要通过一个维度就管理好我的依赖
    这里有个问题要说明，也就是上面的第二点，你肯定会有个疑问，配置里面有那么多的bean，难道很多我不需要的也要每次都加载？是不是应该我没有引入的starter就不应该被加载到？

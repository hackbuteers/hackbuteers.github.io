第一眼看过去，发现是运行时加载了2.5.6版本的ReflectionUtils.findMethod导致，因为它的实现里面while循环直接进不去（传进来的就是Object.class），直接返回null。
本来以为问题到此结束了，但是回过头来看了下参数，发现不对——老版本的方法第三个形参明明是Class[]类型的，但是调用的地方传了一个Object.class，应该报NoSuchMethodError才对啊！这里怎么没有抛异常，还是说把异常吃掉了？想不清楚，直接debug启动过程吧。刚开始的时候，在老版本的ReflectionUtils.findMethod里面打断点，但是不知道为什么一直没进来，因此更加坚信了此处NoSuchMethodError被吞掉了。
上google和stackoverflow上查了很久，并没有查到类似的问题；没办法只好自己模拟，本地写了几个类，编译好之后，修改其中一个类方法的参数，再单独编译这个类（模拟jar包冲突的情况），运行的时候直接抛了NoSuchMethodError…此路不通！
这个时候开始怀疑是类加载器的原因，因为日常部署是部署在tomcat容器中的，应用代码是通过WebAppClassLoader这个类加载器加载的，会不会是这个类加载器加载类的时候修改了类初始化过程，将中抛出来的异常给捕获了。于是翻看WebAppClassLoader的代码，并没有发现这个逻辑（实际上也做不到这样，类初始化过程是JVM调用的）。
此处卡了很久。。。

SerializableTypeWrapper.class反编译出来的代码片段如下：

看到这里才恍然大悟，原来，编译打包时用的是新版本的ReflectionUtils类，它的findMethod方法的第三个参数是可变参数，因此调用的地方实参被编译成数组了，运行时加载了另一个版本的ReflectionUtils，它的方法参数是数组，正好是兼容的，因此没有抛异常，方法体返回了null。

2、总结：

纠结了大半天时间，终于查清问题的根本原因了，算是有点收获~

  • 之前遇到jar包冲突总是伴随着NoClassDefError或者NoSuchMethodError，而这次遇到的情况是在其他地方抛了个NPE，而直接调用的地方方法签名居然是兼容的，只是方法体的逻辑不同，这种情况非常具有迷惑性（如果不是抛NPE，而是计算得到另一个值，将会非常难查）
  • 对可变参数和数组参数的区别有点进一步了解
  • 重温了类初始化过程
